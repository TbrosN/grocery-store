Please implement the following plan to make a sqlite database:
The objective is a local system that ingests grocery product data, normalizes a minimal set of fields, computes health-related flags, and allows very fast filtering to eliminate unhealthy products. The design prioritizes **high recall for healthy items**, aggressive filtering of clearly unhealthy items, and simplicity of ingestion.

The architecture consists of four layers:

1. Raw ingestion storage
2. Normalized product data
3. Parsed ingredient representation
4. Derived health flags for fast queries

The database should be SQLite. The dataset size (likely <1–2M products) is well within SQLite’s capabilities if indexes are used properly.

---

Database Schema

Raw scraped data is preserved so parsing rules can be re-run later.

raw_products
id INTEGER PRIMARY KEY
source_store TEXT
source_url TEXT
scraped_at DATETIME
raw_json TEXT

This table is append-only.

---

Products table holds normalized product metadata.

products
id INTEGER PRIMARY KEY
name TEXT
brand TEXT
category TEXT
organic_flag INTEGER
ingredient_text TEXT
ingredient_count INTEGER
created_at DATETIME

Notes

* ingredient_text stores the original ingredient list string
* ingredient_count is derived during parsing

---

Nutrition facts table contains only the fields relevant to filtering.

nutrition
product_id INTEGER PRIMARY KEY
serving_size_g REAL
calories REAL
protein_g REAL
total_fat_g REAL
saturated_fat_g REAL
trans_fat_g REAL
carbs_g REAL
fiber_g REAL
total_sugars_g REAL
added_sugars_g REAL
sodium_mg REAL

product_id is both PK and FK to products.

---

Ingredients should be normalized only enough to detect problematic ingredients.

ingredients
id INTEGER PRIMARY KEY
canonical_name TEXT UNIQUE

product_ingredients
product_id INTEGER
ingredient_id INTEGER
position INTEGER

PRIMARY KEY (product_id, ingredient_id)

This stores the ingredient list in normalized form.

---

Ingredient classification drives health filtering.

ingredient_flags
ingredient_id INTEGER PRIMARY KEY
is_added_sugar INTEGER
is_artificial_sweetener INTEGER
is_hydrogenated_oil INTEGER
is_seed_oil INTEGER
is_preservative INTEGER
is_junk_additive INTEGER

Only ingredients that appear in your “avoid list” need entries here.

---

Derived health flags table is the most important for query speed.

product_health
product_id INTEGER PRIMARY KEY
contains_added_sugar INTEGER
contains_artificial_sweetener INTEGER
contains_hydrogenated_oil INTEGER
contains_seed_oil INTEGER
contains_preservatives INTEGER
high_added_sugar INTEGER
high_sodium INTEGER
high_trans_fat INTEGER
ultra_processed INTEGER

These fields are computed during ingestion.

Most user queries should only touch this table plus products.

---

Store availability tables.

stores
id INTEGER PRIMARY KEY
name TEXT
website TEXT

store_products
id INTEGER PRIMARY KEY
product_id INTEGER
store_id INTEGER
price REAL
url TEXT
last_seen DATETIME

---

Indexes

Indexes should focus on the fields used for filtering and joins.

products
INDEX idx_products_category(category)

nutrition
INDEX idx_nutrition_added_sugar(added_sugars_g)
INDEX idx_nutrition_sodium(sodium_mg)

ingredients
INDEX idx_ingredient_name(canonical_name)

product_ingredients
INDEX idx_pi_product(product_id)
INDEX idx_pi_ingredient(ingredient_id)

ingredient_flags
INDEX idx_flags_sugar(is_added_sugar)

product_health
INDEX idx_health_filter(
contains_added_sugar,
contains_artificial_sweetener,
contains_hydrogenated_oil,
high_added_sugar,
high_sodium
)

store_products
INDEX idx_store_products_product(product_id)
INDEX idx_store_products_store(store_id)

SQLite will use the composite health index to eliminate most rows quickly.

---

Ingestion Pipeline (Python)

Recommended flow:

1. Scrape product
2. Store raw data
3. Parse nutrition facts
4. Parse ingredient list
5. Map ingredients to canonical names
6. Compute health flags
7. Insert normalized data

Pseudo workflow:

scrape → raw_products
parse → products + nutrition
ingredient parse → ingredients + product_ingredients
flag compute → product_health

---

Ingredient Parsing Strategy

Ingredient lists are messy but workable.

Steps:

1. lowercase text
2. split by commas
3. strip parentheses
4. remove percentages
5. trim whitespace

Example:

"Organic oats, sugar, sunflower oil (high oleic), salt"

Becomes:

organic oats
sugar
sunflower oil
salt

Then attempt lookup in ingredients table.

If ingredient does not exist:

insert canonical entry.

---

Health Flag Computation

During ingestion run SQL queries or Python logic.

Example logic:

contains_added_sugar
if any ingredient_flags.is_added_sugar = 1

high_added_sugar
if nutrition.added_sugars_g > threshold

ultra_processed
if ingredient_count > 12 OR junk_additive present

These results populate product_health.

---

Typical Query

Find healthy products.

SELECT p.name
FROM products p
JOIN product_health h ON p.id = h.product_id
WHERE h.contains_added_sugar = 0
AND h.contains_artificial_sweetener = 0
AND h.contains_hydrogenated_oil = 0
AND h.high_added_sugar = 0
AND h.high_sodium = 0

This query only touches indexed boolean columns.

---

SQLite Configuration

For better performance:

PRAGMA journal_mode=WAL
PRAGMA synchronous=NORMAL
PRAGMA temp_store=MEMORY
PRAGMA foreign_keys=ON

Batch inserts using transactions.

---

Python Implementation Hints

Use sqlite3 or SQLAlchemy.

Prefer batched inserts:

with conn:
cursor.executemany(...)

Avoid inserting one row at a time.

Maintain a small in-memory dictionary mapping ingredient names → ingredient_id to reduce repeated lookups.

Example:

ingredient_cache = {}

If ingredient not in cache:

SELECT id FROM ingredients
INSERT if missing

---

Data Quality Strategy

To avoid false negatives:

* keep blacklist small and conservative
* avoid excluding ingredients unless clearly unhealthy
* rely more on ingredient blacklist than nutrition thresholds

For example:

Exclude:

hydrogenated oils
high fructose corn syrup
aspartame
trans fats

Avoid excluding ambiguous ingredients like “vegetable oil” initially.

---

Future Extensions

The schema supports later additions without major redesign:

* food processing classification (NOVA score)
* micronutrient scoring
* diet filters (keto, vegan)
* personalized rules
* ranking algorithm for healthiest options

No structural change is required to add new health flags.

raw_products

nutrition
ingredients
product_ingredients
ingredient_flags
product_health

Skip store tables until scraping retailers.

This keeps the system simple while enabling the core functionality: **filtering out unhealthy grocery products quickly and reliably**.
